# 批发商城业务全景与排期建议

## 1. 文档目的

本文档用于把当前仓库中的后端项目、管理端项目、批发商城交付文档统一收口为一份可直接用于后续安排的业务排期说明。

适用场景：

- 业务全景梳理
- 后续研发排期
- 前后端联调排期
- 测试与验收分组
- 阶段汇报

本文档重点回答四个问题：

1. 整个平台到底包含哪些业务域
2. 每个业务域的核心流程是什么
3. 这些流程分别落在哪些端和哪些模块
4. 后续排期建议应如何分阶段推进

---

## 2. 项目整体视角

当前项目不是单一后台，而是一个多端协同的商城与批发治理平台。

### 2.1 端侧划分

| 端 | 项目/模块 | 主要职责 |
|---|---|---|
| 平台管理端后端 | `manager-api` | 平台治理、审核、统计、配置、资金、对账、分账、运营、权限 |
| 买家端后端 | `buyer-api` | 用户注册登录、浏览商品、购物车、下单、支付、售后、钱包、营销参与 |
| 商家端后端 | `seller-api` | 商品发布、订单履约、售后处理、采购库存、营销设置、店铺统计 |
| 公共服务后端 | `common-api` | 上传、文件、站点配置、地区、短信、公共资源 |
| IM 服务 | `im-api` | IM 会话、消息、客服问答、坐席配置 |
| 异步消费服务 | `consumer` | 订单、库存、会员、通知、售后、核销、支付等异步事件处理 |
| 平台管理端前端 | `lilishop-admin` | 平台运营后台页面、治理页、统计页、配置页 |
| 业务核心层 | `framework` | 各业务域实体、Service、Mapper、枚举、缓存、ES、MQ、统计聚合 |

### 2.2 核心业务角色

| 角色 | 主要动作 |
|---|---|
| 平台管理员 | 审核店铺、治理代理商、治理商品、治理订单售后、做分账与对账、看经营数据 |
| 店铺/供货商 | 发布商品、发货、处理售后、做营销、管理库存采购、看店铺经营情况 |
| 买家/代理商 | 注册登录、浏览商品、参与活动、下单支付、申请售后、看钱包、做提现 |
| 客服/坐席 | 会话服务、问答维护、IM 处理 |

## 4. 核心业务流程

以下流程是后续排期最应该围绕的主线。

### 4.1 店铺入驻与平台治理流程

流程：

1. 店铺提交入驻申请
2. 状态进入 `SUBMITTED`
3. 平台审核通过或驳回
4. 通过后进入 `APPROVED`
5. 平台可冻结为 `FROZEN`
6. 平台可重新启用回 `APPROVED`

涉及端：

- 商家端：发起申请
- 管理端：审核、冻结、启用、查看日志

后端主落点：

- `manager-api/controller/store`
- `framework/modules/store`

建议优先级：

- `P0`

### 4.2 代理商绑定治理流程

流程：

1. 平台创建代理商身份
2. 创建代理商与店铺绑定申请
3. 绑定申请进入 `SUBMITTED`
4. 平台审核通过进入 `APPROVED + BOUND`
5. 平台审核驳回进入 `REJECTED`
6. 平台可执行解绑进入 `UNBOUND`

涉及端：

- 管理端为主
- 买家端后续承接代理商业务查询与对账

后端主落点：

- `manager-api/controller/agent`
- `framework/modules/agent`

建议优先级：

- `P0`

### 4.3 商品治理与批发运营流程

流程：

1. 商家创建商品、SKU、图片、参数、库存
2. 商品进入审核或直接通过
3. 平台在商品治理页审核、上下架
4. 商品参与批发、积分、卡券等运营语义
5. 首页分类配置决定买家端入口曝光

涉及端：

- 商家端：发布和维护商品
- 管理端：审核、上下架、分类、首页分类、积分/卡券治理
- 买家端：浏览、查询、参与活动

后端主落点：

- `framework/modules/goods`
- `framework/modules/promotion`
- `framework/modules/page`

建议优先级：

- `P0`

### 4.4 买家下单与履约流程

流程：

1. 买家浏览商品或批发商品
2. 进入购物车或直接结算
3. 创建订单，状态为 `UNPAID`
4. 支付成功，进入 `PAID`
5. 进入普通物流、自提或核销履约分支
6. 发货或核销完成后进入 `COMPLETED`

分支：

- 普通物流：`PAID -> UNDELIVERED -> DELIVERED -> COMPLETED`
- 自提：`PAID -> STAY_PICKED_UP -> COMPLETED`
- 核销：`PAID -> TAKE -> COMPLETED`

涉及端：

- 买家端：下单、支付、确认
- 商家端：发货、核销
- 管理端：治理查询、备注、改价、取消、物流查询

后端主落点：

- `framework/modules/order`
- `framework/modules/payment`
- `framework/modules/logistics`

建议优先级：

- `P0`

### 4.5 售后治理流程

流程：

1. 买家提交售后申请，进入 `APPLY`
2. 平台审核
3. 若仅退款且原路退款，可直接完成
4. 若退货退款，进入 `PASS`
5. 买家寄回，进入 `BUYER_RETURN`
6. 商家确认收货后，平台完成退款或终止
7. 最终进入 `COMPLETE`、`REFUSE` 或 `SELLER_TERMINATION`

涉及端：

- 买家端：申请与查看进度
- 商家端：确认收货与处理
- 管理端：审核、退款、查看物流

后端主落点：

- `framework/modules/order/aftersale`
- `manager-api/controller/order`
- `seller-api/controller/order`
- `buyer-api/controller/order`

建议优先级：

- `P0`

### 4.6 核销与异常核销流程

流程：

1. 订单进入待核销状态
2. 商家或平台发起核销
3. 记录核销成功或失败
4. 若 `FAIL` 且 `remark` 非空，纳入异常核销
5. 管理端统一看核销台账和异常汇总

涉及端：

- 商家端：执行核销
- 买家端：看核销记录
- 管理端：核销治理与异常治理

后端主落点：

- `framework/modules/verification`

建议优先级：

- `P1`

### 4.7 营销活动流程

流程：

1. 平台或商家创建营销活动
2. 活动绑定商品
3. 买家领取或参与活动
4. 活动结果与订单、库存、价格联动

重点子流程：

- 拼团：建团、参团、成团、状态启停
- 优惠券：发放、领取、使用
- 满减：活动配置、订单命中
- 积分商品：积分换购
- 卡券商品：商品绑定优惠券能力

后端主落点：

- `framework/modules/promotion`

建议优先级：

- `P1`

### 4.8 钱包、提现与分账流程

流程：

1. 买家或代理商拥有钱包账户
2. 发起充值或提现
3. 平台审核提现
4. 订单完成后进入账单或分账链路
5. 平台按分账规则生成分账记录
6. 店铺核对账单
7. 平台完成结算

对账并行流程：

- 采购对账：围绕采购业务账单
- 资金对账：围绕钱包/资金业务账单

后端主落点：

- `framework/modules/wallet`
- `framework/modules/store`
- `framework/modules/agent`
- `framework/modules/procurement`

建议优先级：

- `P1`

### 4.9 平台经营驾驶舱流程

流程：

1. 后端聚合店铺、商品、订单、分账、核销、代理商数据
2. 管理端工作台统一展示治理总览
3. 再下钻到经营概览、排行、订单概览等统计页

后端主落点：

- `framework/modules/statistics`
- `framework/modules/agent`
- `framework/modules/store`
- `framework/modules/order`
- `framework/modules/wallet`

建议优先级：

- `P2`

## 6. 排期建议

建议不要按代码模块排，而要按业务闭环和依赖关系排。

### 6.1 阶段 A：基础运营闭环

目标：

- 让平台具备最基本可运营能力

建议纳入：

- 登录权限
- 后台账号、角色、菜单
- 店铺入驻审核
- 店铺管理
- 代理商治理
- 店铺审核历史
- 商品管理
- 分类管理

验收标准：

- 平台可以审核店铺
- 可以创建代理绑定关系
- 可以治理商品上下架

建议优先级：

- `P0`

### 6.2 阶段 B：交易履约闭环

目标：

- 让“下单到完成”形成完整链路

建议纳入：

- 购物车与结算联调
- 订单管理
- 发货与部分发货
- 售后管理
- 核销记录

验收标准：

- 至少一条普通物流订单可完成
- 至少一条售后单可走完
- 至少一条核销单可走完

建议优先级：

- `P0`

### 6.3 阶段 C：批发差异化运营闭环

目标：

- 补齐批发项目的差异化业务入口

建议纳入：

- 首页分类配置
- 积分商品治理
- 卡券商品治理
- 批发商品治理
- 异常核销治理

验收标准：

- 买家端可看到配置后的首页分类入口
- 商品可按积分/卡券/批发语义被治理

建议优先级：

- `P1`

### 6.4 阶段 D：营销与增长闭环

目标：

- 让平台具备增长玩法

建议纳入：

- 拼团治理
- 优惠券治理
- 满减治理
- 秒杀与砍价复用能力收口

验收标准：

- 至少一类营销活动可完整建模、上架、参与、成交

建议优先级：

- `P1`

### 6.5 阶段 E：资金与对账闭环

目标：

- 让平台具备财务治理能力

建议纳入：

- 钱包账户
- 提现审核
- 资金流水
- 分账规则
- 分账明细
- 分账治理概览
- 采购对账
- 资金对账

验收标准：

- 提现审核可走通
- 分账规则和记录可查询
- 采购/资金对账可分页、看汇总、看详情

建议优先级：

- `P1`

### 6.6 阶段 F：经营驾驶舱与运营支撑

目标：

- 让平台具备管理驾驶舱和运营收口能力

建议纳入：

- 平台工作台
- 平台经营概览
- 门店排行
- 商品/品类排行
- 订单概览
- 内容管理
- 消息通知
- 平台配置
- 系统支撑与操作留痕

验收标准：

- 平台首页可直接展示治理总览
- 运营和配置项可在后台集中管理

建议优先级：

- `P2`

4. 售后与退款

- 汇总

## 9. 当前建议结论

如果目标是尽快进入可业务试运行状态，建议顺序如下：

1. 先收口 `P0`：店铺、代理商、商品、订单、售后
2. 再收口 `P1`：首页分类、卡券、积分、核销异常、分账、对账
3. 最后补强 `P2`：驾驶舱、内容、消息、系统设置、统计页细节

如果目标是按客户验收口径推进，建议顺序如下：

1. 平台工作台
2. 店铺治理
3. 代理商治理
4. 订单治理
5. 售后治理
6. 核销治理
7. 分账与对账治理
8. 营销与统计补强
