# 批发商城后端与管理端需求分析

## 1. 文档目的

本文档用于在项目开发前，对 `批发商城` 的后端与管理端需求进行一次统一收敛，目标是把以下几类信息打通：

- 原型图里的页面与交互
- 业务角色与权限边界
- 现有后端工程能力与缺口
- 后续接口设计需要的字段
- 管理端需要具备的治理与统计能力
- 项目既有技术规范、接口规范、Swagger 规范

本文档是需求分析文档，不直接替代接口文档或数据库设计文档；但文中会把页面字段、交互逻辑、状态与校验规则整理到足够支撑后续接口与表结构设计的粒度。

## 2. 项目背景与范围

### 2.1 项目背景

当前项目并不是单纯的普通 B2C 商城，而是一个围绕农副产品/批发交易场景构建的多角色商城平台。  
根据原型图，平台需要同时服务以下主体：

- 供货商：提供商品、管理库存、营销、发货、查看收益
- 代理商：理解为区域运营代理，负责交易转化、推广、订单处理、资金结算
- 消费者：浏览商品、下单、参与拼团、核销、自助售后
- 平台后台：审核、治理、分账、统计、拼团、核销、系统配置

### 2.2 本次分析范围

本次只分析：

- 后端服务能力
- 管理端后台能力
- 与后端设计直接相关的前台页面字段和交互逻辑

不展开：

- 前端视觉实现细节
- 小程序/APP 组件实现方式
- 非核心的 UI 样式状态差异

## 3. 现有项目技术与接口规范

### 3.1 技术栈

现有项目是 `Java 21 + Spring Boot 3.5.6 + Maven` 多模块后端工程，主要模块包括：

- `framework`：核心业务实现与公共基础设施
- `manager-api`：管理端接口
- `seller-api`：商家/店铺端接口
- `buyer-api`：买家/前台接口
- `common-api`：公共接口
- `consumer`：MQ、延时任务、事件处理
- `im-api`：IM/WebSocket 服务

主要基础设施：

- MyBatis-Plus
- MySQL + ShardingSphere + Druid
- Redis + Redisson
- RocketMQ
- Elasticsearch
- Spring Security
- SpringDoc OpenAPI

### 3.2 现有接口风格

现有代码中已形成较稳定的后端接口风格：

- 控制器使用 `@RestController`
- 接口路径使用：
  - `/manager/...`
  - `/store/...`
  - `/buyer/...`
  - `/common/...`
- 统一响应包装为 `ResultMessage<T>`
- 分页主流模型为 `PageVO + IPage<T>`
- 控制器普遍使用 `@Validated`
- 接口文档使用 `springdoc-openapi`
- Swagger UI 入口为 `/swagger-ui.html`

### 3.3 开发规范要求

结合仓库级 `AGENTS.md` 与项目现状，后续新增代码应遵守以下原则：

- 新增业务默认按 `Entity / DTO(BO) / VO / Mapper / Service / Controller` 分层
- 不新增裸 JDBC
- 不新增字符串拼 SQL
- 接口继续沿用 `ResultMessage<T>`
- 分页继续兼容 `PageVO`
- 新接口优先返回强类型对象，不继续扩大业务型 `Map<String, Object>` 返回
- 所有新增请求体应补齐 `jakarta.validation` 校验
- 新模块尽量放在 `framework/src/main/java/cn/lili/modules/<domain>`

## 4. 角色体系与权限边界

### 4.1 角色定义

根据原型图，最终后端应支持以下角色：

#### 4.1.1 供货商

核心职责：

- 入驻平台并创建/维护店铺
- 发布商品、管理库存、设置批发阶梯价
- 发起营销活动
- 管理订单、发货、处理售后
- 查看资产、收益、结算、数据概览

#### 4.1.2 代理商

说明：本分析中将代理商理解为区域运营代理角色，是平台核心交易角色。

核心职责：

- 浏览商品、加入购物车、下单、支付
- 管理地址、订单、售后
- 参与拼团、核销
- 查看钱包、提现、资金对账、采购对账
- 查看消息、浏览历史、收藏

#### 4.1.3 消费者

核心职责：

- 浏览商品、下单、支付
- 管理地址、订单、售后
- 参与拼团、核销
- 查看钱包、充值、优惠券、收藏、浏览历史
- 与商家/客服沟通

#### 4.1.4 平台后台管理员

核心职责：

- 审核账号与店铺
- 管理商品、内容、订单、核销、拼团、分账
- 管理角色、权限、系统配置
- 查看平台工作台与数据概览

### 4.2 角色与现有工程映射

现有工程天然已有三类接口面：

- `manager-api`：平台后台
- `seller-api`：店铺/供货商后台
- `buyer-api`：前台用户侧

后续设计时建议映射如下：

- 供货商：优先落在 `seller-api`
- 平台后台：优先落在 `manager-api`
- 代理商与消费者：都落在 `buyer-api`，但需通过角色、菜单、数据权限、接口校验进行区分

### 4.3 权限边界原则

- 供货商不可越权查看平台全量数据
- 代理商不可管理他人店铺
- 消费者不可进入供货商后台或平台后台
- 平台后台可查看全局治理数据，但资金、交易明细要按岗位进一步细分
- 订单、售后、核销、分账数据必须带业务归属约束

## 5. 原型页面总览与业务域拆分

### 5.1 原型页面可归并的业务域

根据图片命名与页面说明，页面可归并为以下业务域：

- 登录注册与角色选择
- 首页与分类
- 商品详情与批发价格
- 购物车与结算
- 地址管理
- 订单与售后
- 拼团
- 核销
- 优惠券与营销
- 钱包、充值、提现、对账
- 消息与聊天
- 收藏与浏览历史
- 供货商工作台与店铺管理
- 商品管理与内容管理
- 平台工作台与数据概览
- 分账管理
- 系统管理与角色管理

### 5.2 重复页面的归并原则

原型中大量页面只是状态差异，不应拆成独立业务需求。例如：

- 订单：待付款、待发货、待收货、待评价、待核销、售后
- 优惠券：可领、未使用、已使用、已过期
- 商品详情：长图、上滑详情、已选规格、结算态
- 购物车：空、普通态、管理态、支付态

后续接口设计时，这些页面应优先通过统一业务实体 + 状态字段解决，而不是为每个状态单独建表或单独建接口集合。

### 5.3 页面说明图的业务规则

原型中的 `页面说明` 图不是普通备注，它补充了页面之间的关键业务规则，后续必须落到后端约束中：

- 底部导航固定为：首页、分类、聊天、团购、批发、我的
- 团购类业务类似拼多多，存在成团人数限制
- 批发业务只改变价格体系，不改变基础商品模型
- 资产/钱包页存在“录入信息后才可提现”的前置条件
- 分账在“售出后进入审核，审核通过自动结算，审核中待结算”的流程下流转
- 核销页存在“核销记录、核销码、立即核销、异常处理”的闭环
- 订单页存在售后、待核销、已核销、自提等业务状态
- 营销管理页包含买赠、满减、指定商品满减、店铺满减等多种活动类型
- 代理商和消费者的页面结构相近，但权限和可操作范围不同
- 拼团成团后才进入正常履约；未成团需自动退款或关闭订单
- 库存需区分锁定与扣减时机：下单锁库存，支付成功扣减，超时未支付释放
- 首页金刚区属于可配置运营位，不应写死在前端

## 6. 核心业务流梳理

### 6.1 登录注册流

原型包含：

- 角色选择
- 本机号码一键登录
- 验证码登录
- 未注册自动注册

后端需要支持：

- 手机号注册/登录
- 短信验证码校验
- 一键登录的第三方能力接入点
- 首次登录后角色绑定
- 角色变更/补充资料入口

### 6.2 供货商入驻流

原型包含：

- 选择店铺类型
- 填写营业执照、统一社会信用代码、地址、有效期
- 上传门头照、店内照、logo
- 录入法人信息、手机号
- 勾选协议

后端需要支持：

- 店铺主体资料提交
- 资质图片上传
- 店铺分类绑定
- 审核状态流转：待提交/待审核/通过/驳回/冻结
- 审核备注记录

### 6.3 下单交易流

前台原型显示代理商与消费者都具备：

- 加购物车
- 订单结算
- 支付结果
- 查看订单

后端需要支持：

- 商品选择、规格选择、数量选择
- 地址选择
- 运费与优惠计算
- 订单提交
- 支付发起
- 支付结果回调
- 订单状态推进

### 6.4 售后流

原型包含：

- 售后申请
- 申请原因
- 申请类型
- 售后详情
- 商家端待处理/已处理

后端需要支持：

- 退款/退货退款等售后类型
- 原因字典
- 售后申请提交
- 售后状态变更
- 商家审核与处理
- 售后日志

### 6.5 拼团流

原型包含：

- 首页精品好团
- 商品详情 “我要开团”
- 拼团详情
- 订单拼团中
- 平台和供货端拼团管理

后端需要支持：

- 拼团活动创建
- 拼团商品关联
- 拼团人数限制
- 拼团有效期
- 拼团状态：未开始/进行中/已结束/已关闭
- 参团记录
- 成团/失败处理
- 拼团订单关联

### 6.6 核销流

原型包含：

- 订单待核销
- 核销详情
- 立即核销
- 平台端核销管理

现有工程已存在“核验码/自提核销”基础接口，但还不是完整核销业务域。  
后端后续需要支持：

- 核销订单查询
- 核销码校验
- 核销执行
- 核销记录查询
- 核销统计
- 核销异常处理

### 6.7 钱包/分账/提现流

原型包含：

- 钱包余额
- 充值
- 提现
- 资金对账
- 采购对账
- 平台端分账规则、分账明细、余额记录

页面说明明确了：

- 商品售出后，分账进入审核
- 审核通过后自动结算到账户
- 可提现金额为已结算金额
- 未录入信息前不能提现

后端需要支持：

- 钱包账户
- 余额变动记录
- 充值申请
- 提现申请
- 分账审核态
- 分账结算态
- 对账单查询

### 6.8 消息与聊天流

原型包含：

- 消息页
- IM 对话入口

现有工程已存在 `im-api` 与 IM 相关模块。  
后续只需补充：

- 角色归属
- 会话对象关系
- 商品卡片/订单卡片等业务消息扩展

### 6.9 数据统计与后台治理流

原型包含：

- 供货商工作台
- 平台工作台
- 平台数据概览
- 店铺/品类排行榜
- 活跃度统计
- 拼团成团率
- 目标指标

后端需要支持：

- 日、周、月、累计维度指标
- 销售额、订单量、活跃用户、活跃店铺
- 分账金额
- 拼团数、成团率
- 核销数、核销率
- 排行榜与导出

## 10. 后续接口与表结构设计原则

后续进入接口设计时，应遵守以下原则：

- 先以页面为单位列接口，再归并为业务模块接口
- 页面字段应能追溯到请求字段/响应字段
- 状态页优先通过统一状态枚举驱动
- 管理端优先设计查询、审核、配置、统计接口
- 前台优先设计列表、详情、提交、状态流转接口
- 核心表结构必须能支撑：
  - 角色隔离
  - 状态流转
  - 审核痕迹
  - 资金流水
  - 核销记录
  - 拼团记录

## 11. 结论

本项目并不需要从零再造一套商城后端，而是要在现有商城后端基础上，围绕原型图做一次“面向批发商城业务”的重整：

- 页面已基本定义了业务流
- 后端需要把角色、资金、核销、拼团、统计重新组织清楚
- 管理端要成为治理中心，而不仅是普通商城后台
- 所有后续接口设计，必须能从页面交互和页面字段反推出来

本文档的下一步承接文档为：

- `批发商城后端接口与表结构设计.md`

## 12. 统一约定

为避免后续接口与页面继续产生口径漂移，统一约定如下：

- 角色名统一使用：
  - `供货商`
  - `代理商`
  - `消费者`
  - `平台后台`
- 页面命名统一按原型口径保留前缀：
  - `代理-`
  - `供货-`
  - `消费-`
  - `平台-` 或 `管理端`
- 相同业务优先只保留一个主名词：
  - `经营概览` 不再同时叫 `工作台/数据概览`
  - `资金对账`、`采购对账`、`我的店铺`、`批发` 作为独立页面名固定使用
- 页面字段、接口字段、表字段保持同名优先：
  - 能用同一字段名表达的，不再拆多个近义字段
  - `region_id`、`order_type`、`shortcutNavList`、`points_goods` 等核心字段应在两份文档里保持一致
- 状态统一由枚举驱动：
  - 订单状态
  - 拼团状态
  - 核销状态
  - 入驻审核状态
  - 分账审核/结算状态
- 页面到接口到表结构的顺序固定为：
  - 页面定义
  - 页面字段
  - 接口清单
  - 请求/响应字段
  - 核心表结构

## 13. 开工前开发约束

正式进入开发前，必须统一以下规则：

- 代码注释统一使用中文，复杂业务流、状态流转、异常分支必须补中文说明
- 数据库表结构必须保留中文备注，字段备注、索引备注、枚举备注都要完整
- 新增字段必须同时补齐：
  - 页面字段说明
  - 接口请求/响应字段
  - 数据库字段备注
  - Swagger 字段说明
- 状态类业务必须先定义枚举，再写接口和表结构
- 任何页面只要存在状态切换、审核、流转、结算逻辑，都必须先写清前置条件、后置条件和不可逆约束

### 验收标准

满足以下条件后，才算具备正式开发条件：

- 需求文档与接口文档无冲突
- 页面字段可以反推接口字段
- 接口字段可以反推表字段
- 表字段都有中文备注
- 代码注释与业务复杂度匹配
- 角色、权限、状态、金额、核销、分账全部有明确约束
