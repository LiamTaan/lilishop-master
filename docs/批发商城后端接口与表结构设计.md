# 批发商城后端接口与表结构设计

## 1. 文档目的

本文档承接《批发商城后端与管理端需求分析》，把业务页面进一步落为：

- 模块级接口清单
- 页面与接口对应关系
- 核心请求/响应字段
- 状态字段与校验规则
- 核心表结构草案

本文档不是最终 Swagger 导出文件，但应足以指导后续：

- 补齐接口设计
- 生成 Swagger 注解
- 设计数据库表
- 拆分开发任务

## 2. 接口规范约定

### 2.1 技术与文档

- 使用现有 `springdoc-openapi`
- Swagger UI 路径：`/swagger-ui.html`
- 每个 Controller 继续使用：
  - `@Tag`
  - `@Operation`
  - `@Parameter`

### 2.2 路径前缀约定

- 平台后台：`/manager/...`
- 供货商后台：`/store/...`
- 前台用户侧：`/buyer/...`
- 公共接口：`/common/...`

### 2.3 统一返回

- 返回体统一为 `ResultMessage<T>`

建议统一字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| success | boolean | 是否成功 |
| message | string | 提示信息 |
| code | int | 业务码 |
| timestamp | long | 时间戳 |
| result | object | 业务结果 |

### 2.4 分页约定

- 入参使用 `PageVO` 或 `SearchParams extends PageVO`
- 返回使用 `IPage<T>`

### 2.5 校验约定

- Controller 使用 `@Validated`
- DTO/BO 使用 `jakarta.validation`
- 路径参数、请求参数补充长度、必填、范围校验

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| applyType | string | 是 | 主体类型 |
| storeType | string | 是 | 店铺类型 |
| storeName | string | 否 | 店铺名称，个体户/企业必填 |
| businessLicenseUrl | string | 否 | 营业执照，个体户/企业必填 |
| creditCode | string | 否 | 社会信用代码，个体户/企业必填 |
| businessLicenseRegionId | string | 是 | 所在地区 |
| businessLicenseExpireType | string | 是 | 固定期限/永久 |
| businessLicenseExpireDate | datetime | 否 | 固定期限时必填 |

| businessHoursStart | string | 否 | 营业开始 |
| businessHoursEnd | string | 否 | 营业结束 |
| realName | string | 否 | 个人主体姓名，个人类型必填 |
| idCardNo | string | 否 | 个人身份证号，个人类型必填 |
| idCardFrontUrl | string | 否 | 身份证正面，个人类型必填 |
| idCardBackUrl | string | 否 | 身份证反面，个人类型必填 |

| legalIdCard | string | 否 | 个体户/企业法人身份证号 |

- `企业法人`：`storeName`、`businessLicenseUrl`、`creditCode`、`legalName`、`legalIdCard`、`legalMobile`
- `企业非法人`：`storeName`、`businessLicenseUrl`、`creditCode`、`legalName`、`legalIdCard`、`legalMobile`、`authorizationUrl`

## 5. Swagger 建设建议

- 一个模块一个 `@Tag`
- 列表、详情、创建、编辑、删除、状态变更接口全部补 `@Operation`
- DTO 字段补 `@Schema`
- 状态字段使用枚举并在文档中说明
- 管理端和前台端路径分开，不混用

- 平台工作台/数据概览
- 核销管理
- 分账管理
- 钱包与提现
- 店铺审核
- 商品发布与批发规则
- 拼团管理

1. 用户角色与登录

## 7. 结论

本项目现有工程并不是空白，而是已经具备：

- 商城基础骨架
- 店铺、商品、订单、拼团、钱包、IM 等部分能力

真正要做的是：

- 把原型图里的页面与交互翻译成可开发的后端接口模型
- 把缺失的核销、分账、资金、管理端统计补齐
- 把页面字段、接口字段、核心表结构统一口径

本文档应与《批发商城后端与管理端需求分析》一起，作为后续正式接口设计、Swagger 补齐和数据库建模的依据。

## 8. 统一约定

为和需求分析文档保持一致，接口与表结构统一按以下规则收口：

- 路径命名统一：
  - 代理商：`/buyer/agent/...`
  - 供货商：`/store/...`
  - 平台：`/manager/...`
  - 公共：`/common/...`
- 一类页面只保留一套主接口：
  - 对账不再保留双路径
  - 我的店铺不再保留双路径
  - 经营概览不再保留双路径
- 入驻申请字段按主体类型条件必填，不再使用一份全类型必填扁平字段集
- 表结构优先复用现有工程的主实体：
  - 商品、订单、拼团、钱包、IM、店铺、积分商品
- 新增表只用于现有表无法表达的业务：
  - 角色关系
  - 代理商对账
  - 金刚区配置
  - 卡券商品
  - 核销记录
  - 分账规则/明细
- 页面说明图中的业务规则需要优先映射到：
  - 状态字段
  - 校验字段
  - 业务流转接口

## 9. 开工前补齐清单

为保证本项目可以一次性进入实施阶段，开发前还需补齐以下工程化约束：

### 9.1 注释与备注规范

- Java Controller、Service、核心业务方法需补中文注释
- 涉及审核、核销、分账、对账、拼团成团等复杂逻辑时，必须补中文业务注释
- Entity 字段、BO 字段、VO 字段应通过注解或注释补足中文说明
- 数据库表、字段、索引、枚举值必须有中文备注
- Swagger `@Schema`、`@Operation` 描述统一使用中文

### 9.2 开工前必须补齐的设计件

1. 角色权限码矩阵
2. 菜单-页面-接口映射矩阵
3. 状态枚举清单
4. 状态流转清单
5. 请求/响应示例报文
6. 金额类字段精度规范
7. 时间字段时区与格式规范
8. 审计字段统一规范

### 9.3 数据库落地要求

- 所有新增表输出建表 SQL 时，必须带中文表注释
- 所有字段必须带中文字段注释
- 状态字段需注明枚举含义，不允许只写“状态”
- 金额字段需注明单位、精度、是否可为空
- 关联字段需注明关联对象，例如“代理商会员 ID”“店铺 ID”“订单号”
- 索引需注明业务目的，例如“按区域+角色唯一”“按订单号查询”

### 9.4 接口落地要求

- 每个页面至少对应一个主列表接口、一个详情接口、一个动作接口
- 每个动作接口必须写明前置条件、状态校验、失败场景
- 列表接口必须统一分页、排序、筛选参数
- 详情接口必须覆盖页面全部展示字段
- Swagger 输出必须可直接给前后端联调使用

### 9.5 推荐实施里程碑

1. 里程碑一：角色、权限、店铺入驻、店铺审核
2. 里程碑二：商品、批发、积分、卡券
3. 里程碑三：订单、发货、售后、核销
4. 里程碑四：拼团、营销、优惠券
5. 里程碑五：钱包、分账、提现、采购/资金对账
6. 里程碑六：经营概览、统计、平台治理、Swagger 收口

### 9.6 一次计划完成项目落地的执行原则

- 先冻结命名、状态、权限、金额规则，再进入编码
- 先补 Swagger 和表备注规范，再批量开发接口
- 先做主链路，再做统计、治理、运营增强能力
- 每完成一个业务域，就同步完成：
  - 接口
  - 表结构
  - 中文注释
  - 中文备注
  - Swagger
  - 基础联调验证
