# 批发商城阶段五范围说明

## 阶段来源

- 阶段五对应完整计划中的“钱包、分账、提现、采购/资金对账”
- 里程碑对应：`里程碑5 = 资金与对账闭环`

## 当前复用底座

- `modules/wallet`
- `modules/store`
- `modules/order/order`
- `modules/procurement`
- `modules/distribution`

## 当前已存在可直接承接能力

- 买家端钱包查询与提现：
  - `GET /buyer/wallet/wallet`
  - `GET /buyer/wallet/wallet/withdrawalSettingVO`
  - `POST /buyer/wallet/wallet/withdrawal`
  - `GET /buyer/member/withdrawApply`
  - `GET /buyer/wallet/log`
- 平台端钱包与提现审核：
  - `GET /manager/wallet/wallet`
  - `PUT /manager/wallet/wallet/increase`
  - `GET /manager/wallet/withdrawApply`
  - `POST /manager/wallet/withdrawApply`
  - `GET /manager/wallet/recharge`
  - `GET /manager/wallet/log`
- 店铺端结算单与流水：
  - `GET /store/order/bill/getByPage`
  - `GET /store/order/bill/get/{id}`
  - `GET /store/order/bill/{id}/getStoreFlow`
  - `GET /store/order/bill/{id}/getDistributionFlow`
  - `PUT /store/order/bill/check/{id}`
  - `GET /store/order/bill/downLoad/{id}`
- 采购底座：
  - `modules/procurement` 已具备采购单、入库单、盘点、报损等强类型模型与服务

## 本阶段建议分两轮推进

### 第一轮：复用增强

- 以现有钱包、提现、结算单能力为主，补齐批发商城文档、联调清单与角色口径。
- 明确代理商在 `buyer-api` 下复用钱包与提现能力，不单独新建钱包体系。
- 平台提现审核继续复用 `manager/wallet/withdrawApply` 主链路。
- 店铺侧账单、流水、对账先复用 `Bill` 与 `StoreFlow` 主链路。

### 第二轮：新增批发差异

- 代理商采购对账列表/详情
- 代理商资金对账列表/详情
- 平台分账规则与分账明细治理
- 代理商经营角色与采购/资金数据隔离联动

## 本轮优先承接范围

- 阶段五文档冻结
- 钱包/提现/平台审核/店铺结算单的联调口径统一
- 代理商对账统一入口已落地：
  - `/buyer/agent/procurement/reconciliation`
  - `/buyer/agent/fund/reconciliation`
  - `/manager/profitsharing/*`

## 本轮已落地

- 代理商采购对账分页、详情、汇总
- 代理商资金对账分页、详情、汇总
- 平台分账治理概览、记录、余额
- 平台分账规则增删改查与建表脚本
- 管理端联调字段映射说明已补齐

## 管理端页面命名口径

- `分账规则`：对应 `/manager/profitsharing/rule`
- `分账明细`：对应 `/manager/profitsharing/record`
- `分账治理概览`：对应 `/manager/profitsharing/balance`
- `采购对账`：管理端后续承接采购账单治理页，不与代理商侧采购对账页混名
- `资金对账`：管理端后续承接资金账单治理页，不与代理商侧资金对账页混名

说明：

- 管理端页面名优先使用“规则 / 明细 / 概览 / 对账”这组稳定词，不再混用“分账管理 / 分账记录 / 平台对账中心”等漂移名称。
- 若一个页面只展示平台聚合余额，不单独命名为“对账页”，统一归为“分账治理概览”。

## 本轮暂不承接

- 新支付通道扩展
- 实际第三方分账打款闭环
- 额外支付网关重选型
